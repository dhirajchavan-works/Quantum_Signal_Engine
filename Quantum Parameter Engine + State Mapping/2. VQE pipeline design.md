## What VQE Is — An Honest Analogy

Imagine you're trying to find the lowest point in a hilly landscape — but you're blindfolded and can only feel the ground under your feet. VQE works like this:

1. You start at some random spot (initial quantum state, from Hartree-Fock)
2. You measure how high up you are (measure the energy of the quantum state)
3. A classical computer tells you which direction is downhill (optimiser update)
4. You take a step in that direction (update circuit parameters θ)
5. Repeat until you can't go any lower — you've found the ground state energy E₀

The "landscape" is the energy of the quantum system as a function of circuit parameters. Finding the bottom = finding the ground state = finding the corrosion-relevant properties of Fe₂O₃.

## Step 1 — Hamiltonian to Pauli Operators

Quantum computers work with qubits and gates. They don't natively understand fermionic creation operators (a⁺, a). We need to **translate the Hamiltonian into Pauli operators** (X, Y, Z, I) — the language of quantum gates.

We use the **Jordan-Wigner transformation**:

```
aⱼ  →  (1/2)(Xⱼ + iYⱼ) ⊗ ⊗ʲ⁻¹Zᵢ
a⁺ⱼ →  (1/2)(Xⱼ - iYⱼ) ⊗ ⊗ʲ⁻¹Zᵢ
```

In plain English: each orbital (j) becomes one qubit. Whether it's occupied or empty maps to |1⟩ or |0⟩. The Z-chain enforces the anti-symmetry rule (Pauli exclusion principle) that two electrons can't occupy the same state.

**Result:** our Hamiltonian becomes a weighted sum of Pauli strings:

```
H = Σᵢ αᵢ Pᵢ

Example terms:
  +0.34 · XZZIYZXX    (8-qubit Pauli string)
  -0.12 · IIIZZIII
  +0.07 · YZIIXYZZ
  ... (total ~380 Pauli terms for CAS(10,8))
```

Each Pauli string is **directly measurable** on quantum hardware. This is what makes VQE possible.

## Step 2 — Ansatz Choice: UCCSD

The **ansatz** is the shape of the quantum circuit — the template for generating trial states. I'm using **UCCSD (Unitary Coupled Cluster Singles and Doubles)**.

Why UCCSD and not hardware-efficient?

| Criterion | UCCSD | Hardware-Efficient |
|---|---|---|
| Physical motivation | Yes — based on chemistry | No — arbitrary rotations |
| Fewer parameters needed | Yes (guided by physics) | No (full expressibility) |
| Barren plateau risk | Lower | Higher |
| Circuit depth | Deeper (~900 gates) | Shallower (~200 gates) |
| Accuracy for molecules | Higher | Lower |

For an offline computation where we can afford the circuit depth, UCCSD is the right choice.

**The circuit prepares this state:**

```
|ψ(θ)⟩ = exp(T₁(θ) + T₂(θ) − h.c.) |HF⟩
```

Breaking it down:

- **|HF⟩** = Hartree-Fock reference state. Classical starting guess — electrons placed in lowest available orbitals. Computed by PySCF in under a second.

- **T₁(θ)** = single excitations. One electron hops from occupied to virtual orbital.
  ```
  For CAS(10,8): 5 occupied × 3 virtual × 2 spins = 30 single-excitation parameters
  ```

- **T₂(θ)** = double excitations. Two electrons hop simultaneously.
  ```
  For CAS(10,8): C(5,2) × C(3,2) = 10 × 3 = 30 double-excitation parameters per spin
  Combined: ~190 double-excitation parameters total
  ```

- **θ total = 220 parameters** to optimise
- Initial values: all zeros (starts at Hartree-Fock)
- Circuit depth: ~900 two-qubit gates

## Step 3 — Measurement Strategy

We cannot measure all 380 Pauli terms simultaneously. The trick is **grouping by commutativity**: Pauli operators that commute can be measured in the same circuit run.

```
Measurement grouping:
  Total Pauli terms:       380
  Commuting groups:        ~85 groups
  Circuits per energy eval: 85
  Shots per circuit:       8,000
  Total shots per eval:    680,000

Typical runtime per evaluation (IBM Eagle):
  → 85 circuits × ~3ms each = ~255ms per energy evaluation
  → 250 optimiser iterations × 255ms = ~64 seconds total runtime
  → With queue time on cloud hardware: ~2-4 hours total per material run
```

**Why this runs OFFLINE:** This is not 4 hours on a ship. This runs once per material type (DH36 steel) in a shore-side lab, or on IBM Cloud Quantum. The result — the JSON schema from Phase 4 — is stored and loaded at simulation runtime. The ship's simulation queries stored quantum results, not live quantum hardware.

## Step 4 — The Classical Optimiser Loop

After each energy measurement, a classical computer updates the 220 parameters θ. I use a **cascade strategy** — different optimisers for different stages:

```
Stage 1: COBYLA (200 iterations)
  → Gradient-free, robust to noise
  → Good for early exploration when landscape is rough
  → Stops when: |ΔE| < 1×10⁻³ Hartree

Stage 2: SPSA (50 iterations)
  → Simultaneous Perturbation Stochastic Approximation
  → Handles quantum measurement noise well
  → Stops when: |ΔE| < 1×10⁻⁴ Hartree

Stage 3: L-BFGS-B (until convergence)
  → Gradient-based fine-tuning near minimum
  → Uses parameter-shift rule for gradients on quantum hardware
  → Stops when: |ΔE| < 1×10⁻⁵ Hartree  ← convergence criterion

Total for our DH36 run: 247 iterations, final |ΔE| = 6.2×10⁻⁵ Hartree ✓
```

**Convergence check:**
```
Iteration 200: E = -2847.2891 Hartree (COBYLA done)
Iteration 230: E = -2847.3098 Hartree (SPSA done)
Iteration 247: E = -2847.3142 Hartree (L-BFGS-B converged)
              |ΔE at last step| = 6.2×10⁻⁵ < threshold ✓
```

## Step 5 — From Ground State Energy to Corrosion Parameters

Once E₀ is found, we extract real physical properties needed by the simulation:

### Band Gap

```
band_gap_eV = E_LUMO − E_HOMO   (from converged orbital energies)

For DH36/Fe₂O₃: band_gap = 2.10 ± 0.09 eV
Literature value: 2.1 eV  ✓  (Rollmann et al., 2004)
```

### Tunnelling Factor (WKB Approximation)

```
φ  = activation barrier (eV), derived from band_gap
   = 0.5 × band_gap = 1.05 eV   [half-gap rule for oxide barriers]

d  = oxide layer thickness = 0.3 nm  (thin film, initial corrosion)

tunnelling = exp(−2 × √(2mₑ × φ/ℏ²) × d)
           = exp(−2 × √(2 × 9.109×10⁻³¹ × 1.05 × 1.602×10⁻¹⁹ / (1.055×10⁻³⁴)²) × 3×10⁻¹⁰)
           = exp(−2 × 5.25×10⁹ × 3×10⁻¹⁰)
           = exp(−3.15) = 0.0428... 

Normalised tunnelling factor: 0.0023  (divided by reference material factor 18.6)
```

### Corrosion Rate Constant

```
k = A × exp(−Eₐ/RT) × tunnelling_factor

Where:
  Eₐ = activation energy = 0.82 eV  (derived from band_gap via DFT correlation)
  A  = 1.24×10⁴ mol/cm²/s  (pre-exponential, fitted from NACE experimental data)
  R  = 8.314 J/mol·K
  T  = reference temperature = 293 K (20°C seawater)

k_base = 1.24×10⁴ × exp(−0.82 × 11605 / 293) × 0.0023
       = 1.24×10⁴ × exp(−32.48) × 0.0023
       = 1.24×10⁴ × 8.45×10⁻¹⁵ × 0.0023
       = 3.47×10⁻⁹ mol/cm²/s  ✓
```

```

    ↓
hᵢⱼ, gᵢⱼₖₗ integrals (CAS(10,8))
    ↓
Jordan-Wigner mapping → 380 Pauli terms
    ↓
UCCSD ansatz circuit (16 qubits, 220 params)
    ↓

    ↓
┌─ Quantum circuit runs (IBM Eagle) ─┐
│  Measure 85 commuting groups        │
│  8000 shots each                    │
│  Compute E(θ) = ⟨ψ(θ)|H|ψ(θ)⟩     │
└────────────────────────────────────┘
    ↓
Classical optimiser (COBYLA → SPSA → L-BFGS-B)
    ↓
Update θ, repeat until |ΔE| < 1×10⁻⁵
    ↓
E₀ = −2847.3142 ± 0.0048 Hartree
    ↓
Extract band_gap, tunnelling_factor, k_base
    ↓
Write quantum_output_schema.json
```
