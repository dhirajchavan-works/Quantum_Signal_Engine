## Why a Formal Contract?

In real engineering systems, two components built by different people need an explicit written agreement about how they communicate. If I change my output format and Kanishk doesn't know, his engine breaks silently — and nobody catches it until a real ship triggers a wrong maintenance alert.

This contract defines:
- Exactly what my simulation sends to the deterministic engine
- Exactly what format it's in
- Exactly what the engine sends back
- Exactly what happens when things go wrong

**Both sides must agree to this contract before any coding begins.** Changes require mutual sign-off.

### Field Definitions

| Field | Type | Unit | Description |
|---|---|---|---|
| `delta_corrosion_mm` | float | mm | Corrosion depth increase this timestep |
| `delta_coating_mm` | float (negative) | mm | Coating thickness decrease this timestep |
| `delta_roughness_um` | float (positive) | μm | Surface roughness increase this timestep |
| `delta_fouling_coverage` | float [0,1] | fraction | Fouling area fraction increase |
| `delta_fouling_thick_mm` | float | mm | Average fouling thickness increase |
| `ci_95` | [float, float] | same as value | 95% confidence interval [lower, upper] |
| `confidence_flag` | string | — | NOMINAL / LOW / CRITICAL |

### Rules for Kanishk's Engine (Mandatory)

1. **Always check `confidence_flag` before using `value`**. If flag = CRITICAL, do not make automatic decisions — return a human-review request.

2. **`delta_*` fields are incremental, not absolute.** The engine must maintain its own running absolute state by accumulating these deltas.

3. **`absolute_state` is advisory** — it reflects Dhiraj's running total. The engine may use it for drift-correction cross-checks but should not override its own state with it without flagging a discrepancy.

4. **Units are fixed** (see table above). No conversion should be applied. If a future version changes units, the `contract_version` will increment.

5. **If `sensor_status` shows anything other than "OK"**, the affected delta values will have wider confidence intervals. The engine should note this in its decision log.

### Acknowledgment Codes

| Code | Meaning | Dhiraj's layer should do |
|---|---|---|
| `ACCEPTED` | Packet processed, state updated | Continue normal operation |
| `REJECTED_FORMAT` | Schema error in packet | Log error, send corrected packet |
| `REJECTED_RANGE` | A value outside physical bounds | Flag for investigation, send last valid packet |
| `REJECTED_STALE` | Packet timestamp too old (>2h) | Resend with current timestamp |
| `HOLD_LOW_CONFIDENCE` | Engine suspended auto-decisions | Normal operation, log flag |

## Error Handling — What Happens When Things Go Wrong

### Network/Timeout Failure

```
If no acknowledgment received within 5 seconds:
  → Retry up to 3 times (1s between retries)
  → If still no response: store packet locally, flag as QUEUED
  → On reconnect: flush QUEUED packets in timestamp order
  → If queue exceeds 24 hours of packets: trigger STALE_DATA alert
```

### Sensor Failure

```
If sensor_status = "FAULT" for a field:
  → Use last valid reading + dead-reckoning estimate
  → Increase sigma for that correction factor by 3×
  → confidence_flag → LOW for affected zones
  → Include "sensor_fault": true in packet for that zone
```

### Quantum Schema Missing or Stale

```
If quantum_output_schema.json is missing:
  → Halt simulation, emit QUANTUM_SCHEMA_MISSING error
  → Do NOT substitute fallback values — wrong k is worse than no k

If quantum schema version doesn't match contract version:
  → Emit SCHEMA_VERSION_MISMATCH warning
  → Downgrade to last compatible version if available
  → Flag Kanishk's engine to expect legacy format
```
